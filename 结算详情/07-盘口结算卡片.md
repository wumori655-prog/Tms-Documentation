### 7.2.1 注单层级新增字段【主队比分】【客队比分】(v1.1 新增)

**字段定义**:

| 字段 | 数据类型 | 取值规则 | 说明 |
|------|---------|---------|------|
| 主队比分 | 整数 | 投注当时主队进球数 | 赛前投注默认为 `0`;开赛后按投注瞬间的实时主队比分记录 |
| 客队比分 | 整数 | 投注当时客队进球数 | 赛前投注默认为 `0`;开赛后按投注瞬间的实时客队比分记录 |

**数据规则**:

| 投注阶段 | 主队比分 | 客队比分 | 备注 |
|---------|---------|---------|------|
| 赛前投注 | 0 | 0 | 比赛尚未开始,统一记录为 0 |
| 滚球投注(开赛后) | 实时主队比分 | 实时客队比分 | 按投注瞬间的实时比分记录,比赛进球后变化 |
| 全场 0:0 结束 | 0 | 0 | 真实数据,不是"无数据"(即便赛前/滚球都是 0/0,仍如实记录) |

**展示位置**:所有渲染器表格在【投注额】列后插入这 2 列(参见 7.3.1 ~ 7.3.4 各章节表格)。

**字段作用**:
- 配合"基准分组"概念使用(参见 7.3.6 自适应基准分组结算规则)
- 主要用于分辨【是否滚球单】(赛前 0/0 vs 滚球真实比分)
- 注意:**该页面不做"判断是否滚球"的逻辑**,仅做数据展示。"是否滚球单"的判断由别的页面(注单详情等)通过【投注时间】字段做。

**🔔 备注**:UI 原型中的【🔄 演示空态切换】按钮无需开发实装,仅用于产品/测试演示效果,真实环境由数据驱动展示。

>
> **撤销原因(v1.1,Mori)**:
>
> 1. **设计上过于强硬**——"自动填入并锁定 B"完全剥夺用户操作灵活性,无法应对特殊场景(如数据错误临时手动覆盖、需要选取消/退款的异常处理等)

> 3. **覆盖范围更广**——原"配对联动"仅适用让球/大小类(MultiLineTable),新机制覆盖所有玩法(让球/大小/独赢/波胆/总进球等),逻辑统一,降低开发与结算员认知负担

## 7.3.6 自适应基准分组结算规则(核心规则)

> 本节是整个结算模块处理"同盘口多组基准分"场景的核心规则。所有相关章节(附录 N7 / 映射表 / 第 9.4 节 / settlement 第 6.1 节/第 6.8 节/第 7.3 节/第 8.1.1 节 / 第 12.15 节)均引用本节。

### 7.3.6.1 背景

滚球让球/大小类盘口的结算基准分 = 终场比分 − 下注时比分(每笔注单独立)。

**关键业务约束**:

- 盘口不支持本地新建盘口线,所有盘口数据来自 IM 数据源
- 当 IM 在比分变化后对该盘口走"暂停-恢复"(同盘口 ID 继续接单)时,同盘口内会出现多笔"下注时比分"不同的注单
- 此时同盘口内按基准分会自然划分为多组,各组结算结果**可能不同且都正确**


**适用范围(硬约束)**:

- ✅ 本节规则**仅适用于常规单场赛事**(EventTypeId = 2)的滚球让球 / 大小类盘口(BT1 / BT2 / BT32 / BT160 / BT161 / BT179 / BT180,及其半场变体)
- ❌ 本节规则**不适用于冠军赛事 / 长期盘口**(EventTypeId = 1),冠军盘口按独立的冠军赛事结算详情模块处理,不涉及"下注时比分"概念
- ❌ 本节规则**不适用于独赢 / 波胆 / 总进球 / 双重机会 / 半全场等不受滚球影响的 BT**(BT3 / BT5 / BT6 / BT7 / BT8 / BT9 / BT158 / BT159),这些 BT 无论赛前或滚球均用终场比分结算,不存在基准分组概念

(后续 7.3.6.2 ~ 7.3.6.7 内容保持不变,详见原文)

## 7.3.7 冲突校验通用规则(v1.1 新增)

> **设计背景**:盘口结算时不同选项的结算结果之间存在业务上的逻辑约束(如让球盘"主队赢"时"客队"不可能也赢)。本节定义所有渲染器类型统一的冲突校验机制,替代原 7.3.1 节"自动配对联动"的强硬填入方案。

### 7.3.7.1 核心原则

**业务铁律**:任何盘口结算结果,**"赢"只能有一个**:
- 让球类两个选项不能同向赢
- 独赢类三选一只能一个赢
- 波胆/总进球多选项只能一个赢

| 渲染器 | 玩法举例 | 冲突判定 |
|---|---|---|
| MultiLineTable | 让球 BT1、大小 BT2 | 两个选项不能同向(同赢/同赢半/同输/同输半/任一含"和") |
| SingleLineTable | 独赢 BT3(主/和/客) | 三个选项最多只能一个"赢",其他必为"输" |
| Matrix | 波胆 BT6、反波胆 BT158、半全场 BT9 等 | 所有选项最多只能一个"赢" |
| LongList | 总进球 BT7、双重机会 BT8 等 | 所有选项最多只能一个"赢" |

### 7.3.7.2 校验时机:提交时校验(非自动联动)

```
用户行为:在盘口结算弹窗中选择每个选项的结算结果
↓
点击「提交结算」/「确认二次结算」按钮
↓
系统校验该盘口所有选项的结算结果组合
↓
检测到冲突:
  弹窗警告(柔性):
    "检测到多个'赢',此玩法只能有一个赢,请确认?"
    [取消修改] [坚持提交]
  
  用户点[取消修改] → 返回结算弹窗,允许修改
  用户点[坚持提交] → 允许提交 ✅ 但强制记日志(详见 7.3.7.4)
```

> **设计意图**:不阻止用户提交(保留特殊场景的人工覆盖能力),但通过弹窗+日志双层防护,既防止误操作又留下审计痕迹。

### 7.3.7.3 校验文案

各类玩法在冲突时展示**统一格式**的警告文案:

| 玩法类型 | 警告文案 |
|---|---|
| 让球/大小(MultiLineTable) | "检测到多个'赢',此玩法只能有一个赢,请确认?" |
| 独赢(SingleLineTable) | "检测到多个'赢',此玩法只能有一个赢,请确认?" |
| 波胆/反波胆/半全场(Matrix) | "检测到多个'赢',此玩法只能有一个赢,请确认?" |
| 总进球/双重机会(LongList) | "检测到多个'赢',此玩法只能有一个赢,请确认?" |

> **设计意图**:文案统一,降低结算员认知负担;各玩法的具体业务逻辑由结算员自身判断,文案不展开。

### 7.3.7.4 强制日志记录规则

以下两类操作必须强制记录到结算日志(归入结算详情第 21 章「结算日志」):

| 触发场景 | 日志类型 | 日志内容必含字段 |
|---|---|---|
| 用户被冲突警告后坚持提交 | warning_override | 操作人 + 时间 + 盘口 ID + BT 类型 + 盘口线 + 各选项的结算结果组合 + "用户被警告但坚持提交"标记 |
| 用户选择"取消"或"退款"作为结算结果 | special_result | 操作人 + 时间 + 盘口 ID + BT 类型 + 盘口线 + 选择(取消 / 退款) |

> **日志归属**:本节定义的两类日志统一归入结算详情第 21 章「结算日志」,与现有结算日志并入同一查询入口。本节不重复定义日志展示与查询规则,只规定记录触发条件与必含字段。

### 7.3.7.5 例外说明(以下组合不触发冲突校验)

以下选项组合**不触发冲突校验**(直接放行;但若选了取消/退款,仍按 7.3.7.4 记日志):

| 选项组合 | 说明 |
|---|---|
| 全部选项都选"取消" | 整个盘口取消,合法 |
| 全部选项都选"退款" | 整个盘口退款,合法 |
| 全部选项都选"走水"(仅 MultiLineTable 整数让分如 -1.0、-2.0) | 整盘走水,合法 |
| 让球/大小 一边赢一边输 / 一边赢半一边输半 | 标准结算,合法 |
| 让球/大小 一边赢半一边走水(部分整数让分场景) | 标准结算,合法 |
| 独赢/波胆/总进球 只有一个赢、其他都选"输" | 标准结算,合法 |
| 独赢/波胆/总进球 所有选项都选"输"(理论极端,实际不可能) | 不触发冲突校验,但需结算员核对赛果数据 |

### 7.3.7.6 与其他校验的关系(v1.1 补充)

本节定义的是**业务冲突校验**——检测结算结果**组合**是否符合业务铁律(如"赢只能有一个")。本节不涉及以下其他校验层:

| 校验层 | 责任 | 归属 |
|---|---|---|
| **业务冲突校验** | 结算结果组合的业务合法性 | **本节 7.3.7** |
| 字段必填校验 | API 必填字段(如结算时间戳、操作人 ID 等) | 后端 API 契约 |
| 字段格式校验 | 数据类型 / 精度 / 取值范围 | 后端 API 契约 |
| 权限校验 | 当前用户是否有该盘口的结算权限 | 第 17 章权限控制 |
| 并发与竞态校验 | 操作版本号 / 防重复提交 | 第 19.3 章 |

**多层校验执行顺序(建议)**:权限校验 → 字段必填校验 → 字段格式校验 → 业务冲突校验 → 并发版本校验。任一层校验失败时立即返回对应错误,不进入下一层。

**重要边界**:本节不约束其他校验层的实装方式;但**所有校验层的失败提示**应遵循 UX 友好原则——不直接暴露后端字段名 / SQL 错误 / 技术细节,统一映射为中文业务提示文案。

## 7.7 操作按钮状态

### 7.7.1 可操盘盘口

| 盘口状态 | 结算按钮 | 二次结算按钮 |
|---------|---------|------------|
| 开盘中 | 不显示(标记"开盘中") | 不显示 |
| 关盘 + 未选择结果 | 禁用 | 不显示 |
| 关盘 + 已选择结果 | 启用(蓝色) | 不显示 |
| 已结算 + 赛事待结算 | 灰色「已结算」不可点击 | 显示「二次结算」按钮(可直接进入二次结算) |
| 已结算 + 赛事非待结算 | 灰色「已结算」不可点击 | 「二次结算」按钮禁用,悬停提示"请先执行赛事驳回" |
| 已取消 | 不显示 | 不显示 |

> 关盘 = 绝对终态,无人工开盘入口

### 7.7.2 数据源盘口

| 盘口状态 | 结算按钮 | 二次结算按钮 |
|---------|---------|------------|
| 数据源接管态(任意状态) | 不显示(只读) | 不显示 |
| 待手动结算(超时解锁) | 启用(本地结算) | 不显示 |
| 本地锁定(已本地结算) | 不显示 | 显示「二次结算」按钮(本地差额处理) |

> 盘口状态完整定义和流转规则见第 18 章。
>
> **赛事级前置限制**:
>
> - **首次结算(关盘→已结算)/ 盘口级取消**:赛事状态 = **待结算 或 异常态**允许(行业做法:异常态下有明确赛果的盘口可正常首次结算,如腰斩时上半场类盘口);已结算 / 已取消时禁用
> - **盘口二次结算(已结算→关盘)**:仅赛事状态 = 待结算;其他状态禁用,鼠标悬停提示"请先执行赛事驳回"
>
> 数据源盘口的按钮展示不受此限制影响(本来基本只读)。
>
> **按钮粒度硬约束**:结算 / 二次结算 / 取消等所有操作按钮均在盘口线级(MultiLineTable)或玩法级(其余渲染器)粒度上展示。**选项级(Matrix 单元格、LongList 单选项、SingleLineTable 选项行、BT6/BT158/BT9 矩阵任一格子)不得出现以上任何按钮**。完整粒度规则见第 18 章 18.2.2 操作按钮粒度规则。

## 7.8 盘口结算前置条件

**只要满足以下两个条件,盘口即可执行结算,不需要等整个赛事完赛**:

| 条件 | 说明 |
|------|------|
| 盘口必须已关盘 | 关盘来源:**数据源推送关盘(唯一来源)**。操盘页与结算页面均不提供关盘入口;关盘 = 绝对终态,无人工开盘入口。详见第 18 章关盘来源 |
| 该盘口所需赛果已确定 | 不同判定时机类型的盘口,所需赛果字段不同,详见 7.8.1 |
| **BT6 波胆除外** | BT6 不需要关盘,赛中比分变化时系统实时扫描已无法中奖的选项标记为输 |

### 7.8.1 盘口按判定时机分类

| 判定时机类型 | 所需赛果 | 典型 BT | 结算时机 |
|-----------|---------|---------|---------|
| **上半场类 (1H)** | 半场比分 + 半场扩展(角球/牌)+ 半场进球详情 | BT1-1H / BT2-1H / BT6-1H / BT9 半全场(半场部分)/ 半场角球类 / 半场牌类 | 半场结束后(上半场弹窗应用时)|
| **下半场类 (2H)**【独立分类】| 下半场比分 + 下半场扩展(角球/牌)+ 下半场进球详情 | BT1-2H / BT2-2H / BT6-2H / BT24 / BT25 / BT26 / BT27 / BT28 / BT29 / BT30 / BT33 / BT34 / BT57 / BT19 / BT20 / BT291 等 / 下半场角球类 / 下半场牌类 | 下半场结束后(下半场弹窗应用时,前置上半场已应用)|
| 15 分钟段类 | 段内比分 | BT48 / BT179 / BT180(15 分钟段)| 该段结束后 |
| **事件触发类** | 进球明细 / 角球明细 / 首个事件时间轴 | BT37 第 X 粒进球队伍 / BT159 角球第 X 粒 | **事件发生即可结算**,可能早于终场 |
| 全场类 (FT) | 全场合计比分(= 上+下)+ 全场累计扩展 + 加时 + 点球大战 | BT1 / BT2 / BT3 / BT5 / BT7 / BT8 / BT158 / BT160 / BT161 等 / 全场角球类 / 全场牌类 / BT9 半全场(全场部分)/ 加时类(分区 B)/ 晋级类(分区 C)| 全场结束后(全场扩展弹窗分区 A / B / C 各自应用时)|
| 比分变动触发类 | 赛中实时比分 | BT6 波胆(赛中判输部分)| 比分变化时触发 |

### 7.8.2 按判定时机独立结算

各盘口按自身判定条件走独立结算流程——半场结束后上半场类盘口可结算,第 3 粒进球发生后相关事件触发类盘口可结算,不必等到全场终场。
